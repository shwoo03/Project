# 프로젝트 개선 워크플로우

> 이 스킬은 모든 프로젝트에 적용 가능한 범용 스킬입니다.

---

## Phase 1: 프로젝트 파악

### 1.1 루트 디렉터리 MD 파일 분석
**목표**: 프로젝트의 목적, 구조, 현재 상태를 파악한다.

**수행 작업:**
1. 루트 디렉터리의 모든 `.md` 파일 목록 확인
2. 다음 파일들을 우선적으로 읽기:
   - `README.md` - 프로젝트 개요, 설치 방법, 사용법
   - `CHANGELOG.md` - 변경 이력
   - `CONTRIBUTING.md` - 기여 가이드
   - `ARCHITECTURE.md` - 아키텍처 설명
   - `ROADMAP.md` - 개발 로드맵
   - `TODO.md` - 할 일 목록
3. 이미 분석한 적 있다면 이 단계 생략

**출력 형식:**
```
📋 프로젝트 요약
- 프로젝트명: [이름]
- 목적: [한 줄 설명]
- 기술 스택: [언어, 프레임워크, 라이브러리]
- 주요 기능: [기능 목록]
- 현재 상태: [개발 중/완료/유지보수]
```

## Phase 2: 개선 제안

### 2.1 소스 코드 분석
**목표**: 코드와 기능을 분석하여 개선점을 도출한다.

**분석 관점:**
1. **성능**: 비효율적인 알고리즘, 불필요한 연산, 메모리 누수
2. **보안**: 하드코딩된 비밀정보, 입력값 검증 부재, 취약한 의존성
3. **로직 결함**: 엣지 케이스 미처리, 예외 처리 부재, 레이스 컨디션
4. **코드 품질**: 중복 코드, 긴 함수, 복잡한 조건문, 명명 규칙
5. **확장성**: 결합도, 모듈화, 인터페이스 설계

### 2.2 제안서 작성
**출력 형식:**

```markdown
## 🔧 리팩터링 제안 (3가지)

### R1. [제목]
- **카테고리**: 성능 / 보안 / 로직 / 코드 품질
- **현재 문제**: [구체적 문제 설명]
- **영향 범위**: [관련 파일/함수]
- **예상 효과**: [개선 시 기대 효과]
- **난이도**: ⭐ / ⭐⭐ / ⭐⭐⭐

### R2. [제목]
...

### R3. [제목]
...

---

## ✨ 신규 기능 제안 (3가지)

### F1. [제목]
- **설명**: [기능 설명]
- **필요성**: [왜 필요한지]
- **구현 방향**: [대략적 구현 방법]
- **예상 효과**: [사용자/시스템 관점 이점]
- **난이도**: ⭐ / ⭐⭐ / ⭐⭐⭐

### F2. [제목]
...

### F3. [제목]
...
```

### 2.3 사용자 선택 대기
사용자에게 R1-R3, F1-F3 중 하나를 선택하도록 요청한다.

---

## Phase 3: 상세 계획 수립

### 3.1 인터넷 리서치
**목표**: 선택된 항목에 대해 최적의 구현 방법을 조사한다.

**리서치 소스:**
- 📄 **논문**: Google Scholar, arXiv
- 📝 **블로그**: Medium, Dev.to, 개인 기술 블로그
- 💻 **오픈소스**: GitHub 유사 프로젝트, 라이브러리 공식 문서
- 📚 **공식 문서**: 언어/프레임워크 공식 가이드
- 💬 **커뮤니티**: Stack Overflow, Reddit, 한국 개발 커뮤니티

**리서치 체크리스트:**
- [ ] 유사한 문제를 해결한 사례 검색
- [ ] 권장되는 디자인 패턴 또는 아키텍처 확인
- [ ] 사용 가능한 라이브러리/프레임워크 조사
- [ ] 잠재적 문제점 및 해결책 파악
- [ ] 성능/보안 벤치마크 자료 수집

### 3.2 구현 계획서 작성
**artifact 경로**: `brain/<conversation-id>/implementation_plan.md`

**계획서 구조:**
```markdown
# [선택된 항목] 구현 계획서

## 1. 개요
- 목표
- 범위
- 예상 소요 시간

## 2. 리서치 결과
- 참고 자료 목록 (URL 포함)
- 핵심 인사이트
- 선택한 접근 방식 및 이유

## 3. 상세 설계
### 3.1 아키텍처 변경사항
- 변경 전/후 다이어그램

### 3.2 파일별 변경 계획
- [NEW/MODIFY/DELETE] 파일명: 변경 내용

### 3.3 데이터 구조/인터페이스
- 타입 정의
- API 인터페이스

## 4. 구현 단계
1. [ ] 단계 1: ...
2. [ ] 단계 2: ...
...

## 5. 테스트 계획
- 단위 테스트 항목
- 통합 테스트 항목
- 수동 검증 항목

## 6. 롤백 계획
- 실패 시 복구 방법

## 7. 참고 문헌
- 논문, 블로그, 오픈소스 링크
```

### 3.3 사용자 검토
계획서를 사용자에게 공유하고 승인을 받는다.

---

## Phase 4: 구현 및 테스트

### 4.1 구현
**원칙:**
1. 계획서의 단계를 순서대로 진행
2. 각 단계 완료 후 중간 검증
3. 기존 기능 회귀 방지
4. 충분한 주석 및 문서화

**코드 작성 가이드:**
- 프로젝트의 기존 코딩 스타일 유지
- 새로운 의존성은 최소화
- 하위 호환성 고려

### 4.2 테스트
**테스트 레벨:**

| 레벨 | 내용 | 통과 기준 |
|------|------|----------|
| 1 | 문법/컴파일 오류 없음 | 빌드 성공 |
| 2 | 단위 테스트 통과 | 모든 테스트 PASS |
| 3 | 기존 기능 회귀 없음 | 기존 테스트 PASS |
| 4 | 신규 기능 정상 동작 | 수동 검증 완료 |
| 5 | 엣지 케이스 처리 | 예외 상황 테스트 |

**실패 시 대응:**
1. 에러 메시지 분석
2. 원인 파악 및 수정
3. 재테스트
4. 3회 이상 실패 시 접근 방식 재검토

### 4.3 반복
모든 테스트가 통과할 때까지 4.1 ~ 4.2를 반복한다.

## Phase 5: 문서 업데이트

### 5.1 업데이트 대상 파일
루트 디렉터리의 MD 파일들을 용도에 맞게 업데이트한다.

| 파일 | 업데이트 내용 |
|------|-------------|
| `README.md` | 새 기능 설명, 사용법, 설정 변경사항 |
| `CHANGELOG.md` | 버전, 날짜, 변경 내용 추가 |
| `ARCHITECTURE.md` | 구조 변경 시 다이어그램/설명 수정 |
| `ROADMAP.md` | 완료 항목 체크, 새 계획 추가 |
| `TODO.md` | 완료 항목 제거 또는 체크 |
| `API.md` | API 변경 시 엔드포인트 문서 수정 |
| `CONTRIBUTING.md` | 새 규칙 추가 시 업데이트 |

### 5.2 CHANGELOG 형식
```markdown
## [버전] - YYYY-MM-DD

### Added
- 새로 추가된 기능

### Changed
- 변경된 기능

### Fixed
- 버그 수정

### Removed
- 제거된 기능

### Security
- 보안 관련 변경
```

### 5.3 최종 검토
- [ ] 모든 관련 MD 파일 업데이트 완료
- [ ] 문서와 코드 일치 여부 확인
- [ ] 오타 및 문법 오류 검수
- [ ] 링크 유효성 확인
